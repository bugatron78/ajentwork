# AJ Technical Implementation Blueprint

## Purpose

This document turns the v1 product spec into an implementation-ready blueprint for `aj`, a local-first CLI work tracker for AI agents with optional Jira interoperability.

This blueprint assumes:

- `aj` is the canonical operational layer for agents
- Jira is an optional linked mirror for human teams
- canonical state is stored in git-friendly files under `.aj/`
- a local SQLite cache exists only to accelerate reads and searches
- the CLI is designed for compact, agent-friendly interaction

## Recommended Stack

### Language

Use Go for v1.

Why:

- single static binary
- fast startup, which matters for CLI-heavy agent workflows
- simple cross-platform distribution
- strong standard library for files, locks, HTTP, and JSON
- good ecosystem for CLI parsing, TOML, and SQLite
- low operational complexity compared to Node or Python

### Go Version

Target the current stable Go release at implementation time and enforce it in `go.mod`.

### Suggested Libraries

Keep dependencies intentionally small.

- CLI parsing: `github.com/spf13/cobra`
- TOML parsing: `github.com/pelletier/go-toml/v2`
- SQLite driver: `modernc.org/sqlite`
- File locking: `github.com/gofrs/flock`
- Text diffing for debug output only: keep optional unless needed

Important: Cobra should only handle argument parsing and dispatch. Help output must come from `aj`'s own structured help registry, not from raw autogenerated flag text.

## Architectural Layers

Use a narrow layered architecture:

1. `cli`
   Parses args, resolves output format, invokes application services.

2. `app`
   Use-case layer. Contains command handlers such as `CreateItem`, `TakeItem`, `SyncJiraItem`, `RenderWorkflowHelp`.

3. `domain`
   Core types and rules. No filesystem, HTTP, or CLI dependencies.

4. `store`
   File-backed canonical persistence for items, events, config, and artifacts.

5. `sqlitecache`
   Read-optimized index and query cache.

6. `jira`
   External adapter for Jira import/export/sync.

7. `help`, `render`, `workflows`, `search`
   Discoverability and output surfaces.

Dependency rule:

- outer layers may depend on inner layers
- inner layers must not depend on outer layers
- `domain` must not import `cobra`, `sqlite`, or Jira code

## Core Domain Model

Primary types:

- `Item`
- `Lease`
- `Event`
- `RelationSet`
- `Acceptance`
- `ExternalLink`
- `SyncState`
- `ArtifactRef`

Suggested core structs:

```go
type ItemID string
type EventID string
type AgentName string

type ItemKind string
const (
    KindBug     ItemKind = "bug"
    KindFeature ItemKind = "feature"
    KindTask    ItemKind = "task"
    KindSpike   ItemKind = "spike"
    KindEpic    ItemKind = "epic"
)

type Status string
const (
    StatusTodo       Status = "todo"
    StatusInProgress Status = "in_progress"
    StatusBlocked    Status = "blocked"
    StatusInReview   Status = "in_review"
    StatusDone       Status = "done"
    StatusCanceled   Status = "canceled"
)

type SyncMode string
const (
    SyncImportOnly    SyncMode = "import_only"
    SyncExportOnly    SyncMode = "export_only"
    SyncBidirectional SyncMode = "bidirectional"
)

type SyncState string
const (
    SyncClean       SyncState = "clean"
    SyncDirtyLocal  SyncState = "dirty_local"
    SyncDirtyRemote SyncState = "dirty_remote"
    SyncConflict    SyncState = "conflict"
)
```

Core invariants:

- `Item.ID` is immutable
- `Item.Status` must be one of the internal statuses
- `NextAction` is required when status is `todo`, `in_progress`, or `blocked`
- only one active lease may exist per item
- every mutating operation writes an event
- current item snapshot must be derivable from events, even though `meta.toml` is canonical for reads

## Storage Model

Canonical layout:

```text
.aj/
  config.toml
  locks/
  issues/
    W-8F3K2P1Q/
      meta.toml
      events/
        2026-03-13T14-02-11Z_01HR...toml
  artifacts/
  cache/
    index.sqlite
```

### Canonical Files

- `.aj/config.toml`
  Global repo configuration including Jira config and status mappings.

- `.aj/issues/<id>/meta.toml`
  Compact current snapshot of an item.

- `.aj/issues/<id>/events/*.toml`
  Append-only event stream.

- `.aj/artifacts/...`
  Optional long notes, logs, exported reports, and other large content.

### Local-Only Files

- `.aj/cache/index.sqlite`
  Read index, query acceleration, search acceleration, and help search cache if needed.

Recommended `.gitignore` entries:

```text
.aj/cache/
.aj/locks/
```

### Write Strategy

For any mutating command:

1. acquire store lock
2. load `meta.toml`
3. validate operation against domain rules
4. append event file
5. rewrite `meta.toml`
6. update SQLite cache
7. release lock

The event append must happen before `meta.toml` rewrite so recovery can detect partially applied writes.

### Locking Strategy

Use file locks through `.aj/locks/`.

Suggested locks:

- repo-level lock for config and cache migrations
- item-level lock for individual item updates

Recommended behavior:

- wait briefly for lock acquisition
- fail with actionable error if lock cannot be acquired
- never spin indefinitely

## SQLite Cache

The cache is a derived index, not a source of truth.

Primary tables:

- `items`
- `leases`
- `relations`
- `events`
- `external_links`
- `artifacts`

Key use cases:

- `aj ls`
- `aj next`
- `aj inbox`
- `aj changes`
- `aj help search`

Cache rebuild strategy:

- bootstrap on `aj init`
- rebuild automatically if missing or schema version mismatches
- support `aj doctor` or `aj cache rebuild` later, but do not require it in v1

## Configuration

Example `.aj/config.toml`:

```toml
schema_version = 1
default_output = "brief"
default_lease_ttl = "4h"

[jira]
enabled = true
base_url = "https://example.atlassian.net"
project = "ABC"

[jira.status_map]
"To Do" = "todo"
"In Progress" = "in_progress"
"Blocked" = "blocked"
"In Review" = "in_review"
"Done" = "done"
```

Authentication should not be stored in repo config.

For v1, read Jira credentials from environment variables only:

- `AJ_JIRA_BASE_URL`
- `AJ_JIRA_EMAIL`
- `AJ_JIRA_TOKEN`

If the environment values are present, they override config values where appropriate.

## ID Generation

Use short random IDs in v1.

Format:

- item IDs: `W-XXXXXXXX`
- event IDs: `EV-XXXXXXXXXXXX`

Requirements:

- URL-safe ASCII
- easy to read aloud
- low collision rate
- no central allocator required

Implementation recommendation:

- generate cryptographically random bytes
- encode with a reduced alphabet chosen for readability
- validate uniqueness on write

## CLI Design

### Root Command

`cmd/aj/main.go` should be very small. It should:

- construct the dependency graph
- build the command tree
- execute
- translate errors into compact terminal output and exit codes

### Global Flags

Keep these few and stable:

- `--repo <path>`
- `--format brief|json|tsv|prompt`
- `--no-color`
- `--verbose`

Avoid a large shared global flag surface.

### Command Tree

Recommended tree:

```text
aj
  init
  new
  ls
  show
  take
  release
  update
  block
  unblock
  handoff
  done
  reopen
  link
  next
  inbox
  changes
  help
  commands
  workflows
  examples
  glossary
  jira
    pull
    push
    link
    sync
    unlink
```

Special-case alias:

- `aj take jira <key> --agent <name>`

This can be implemented as a custom subcommand under `take` rather than hidden argument parsing magic.

### Command Implementation Pattern

Each command should follow a consistent pattern:

1. parse args into an input struct
2. validate local argument semantics
3. call one application service
4. render a typed result

Example:

```go
type TakeInput struct {
    ItemID ItemID
    Agent  AgentName
    TTL    time.Duration
    Force  bool
}

type TakeResult struct {
    Item Item
    Lease Lease
}
```

Avoid commands directly mutating files.

## Help System

The help system is a first-class subsystem.

### Help Registry

Create a structured help registry in `internal/help`.

Suggested types:

```go
type Doc struct {
    Name           string
    Summary        string
    Purpose        string
    Usage          string
    Arguments      []ArgDoc
    Options        []OptionDoc
    Examples       []ExampleDoc
    Related        []string
    Safety         []string
    WorkflowTags   []string
    SearchKeywords []string
}
```

This registry is the single source of truth for:

- `--help`
- `aj help <command>`
- `aj commands`
- `aj workflows`
- `aj examples`
- `aj glossary`
- `aj help search`

### Help Renderers

Implement renderers for:

- `brief`
- `json`
- `prompt`

Rules:

- `brief` is default human-readable output
- `json` is structured and stable
- `prompt` is terse and optimized for direct agent consumption

### Help Search

`aj help search <term>` should search:

- command names
- summaries
- options
- safety notes
- workflow tags
- example labels

The search implementation can start as in-memory scanning over the registry. No heavy indexing is needed in v1.

### Workflows And Examples

Represent workflows and examples as structured records, not freeform blobs.

Suggested types:

```go
type WorkflowDoc struct {
    Name  string
    Topic string
    Steps []string
}

type ExampleSet struct {
    Topic    string
    Examples []ExampleDoc
}
```

This makes it easy to render:

- terminal output
- JSON
- future Markdown docs

## Rendering

Create a render layer that formats typed results.

Recommended renderers:

- item summary renderer
- list row renderer
- prompt renderer
- machine JSON renderer
- help renderer

Do not let application services emit plain strings.

Output design rules:

- defaults must be compact
- no full event history unless explicitly requested
- prompt output must avoid decorative phrasing
- JSON output should use stable field names

## Jira Adapter

The Jira adapter lives behind an interface in `internal/jira`.

Suggested interface:

```go
type Client interface {
    FetchIssue(ctx context.Context, key string) (RemoteIssue, error)
    CreateIssue(ctx context.Context, input CreateIssueInput) (RemoteIssue, error)
    UpdateIssue(ctx context.Context, key string, patch UpdateIssueInput) (RemoteIssue, error)
    AddComment(ctx context.Context, key string, body string) error
}
```

Do not let the rest of the system depend on raw Jira response models.

Create adapter-local translation types:

- `RemoteIssue`
- `RemoteStatus`
- `RemoteComment`

### Jira Service Responsibilities

The application service for Jira should:

- import a Jira issue into an `Item`
- export an `Item` to Jira
- sync status and milestone comments
- detect local/remote divergence
- set `sync_state`
- refuse unsafe silent merges

### Sync Algorithm

For `aj jira sync <id>`:

1. load item and linked Jira metadata
2. fetch remote issue
3. compare remote version to stored `last_remote_version`
4. compare local `updated_at` to `last_synced_at`
5. determine `clean`, `dirty_local`, `dirty_remote`, or `conflict`
6. if clean, do nothing
7. if only remote changed, pull mapped fields
8. if only local changed, push mapped fields and milestone comments
9. if both changed, fail unless `--resolve` was given
10. write sync event and update link metadata

### Milestone Comment Policy

Only emit Jira comments for milestone events:

- claimed
- blocked
- handoff
- ready for review
- done

Do not mirror every local event to Jira.

### Jira Scope For V1

Support Atlassian Cloud only in v1.

Keep Jira field support intentionally narrow:

- title / summary
- description
- labels
- priority
- assignee when available
- status transitions through configured mapping
- comments for milestones

Do not support in v1:

- arbitrary custom fields
- board or sprint operations
- attachment upload
- webhooks
- rich comment threading

## Error Model And Exit Codes

Use typed errors in the application layer and map them to user-facing output in `cli`.

Suggested categories:

- invalid input
- not found
- lease conflict
- validation failure
- sync conflict
- auth failure
- config error
- storage error
- network error

Suggested exit code policy:

- `0` success
- `1` generic failure
- `2` invalid usage
- `3` not found
- `4` lease or sync conflict
- `5` auth/config issue

Error messages should always be recovery-oriented.

Example:

```text
sync conflict: local and Jira changed since last sync
try: aj jira sync W-8F3K2P1Q --dry-run
or:  aj jira sync W-8F3K2P1Q --resolve keep-local
```

## Schema Versioning And Migrations

Add `schema_version` to `.aj/config.toml` and to cache schema metadata.

Migration policy:

- canonical file migrations are explicit and versioned
- SQLite cache is disposable and may be rebuilt on version mismatch
- v1 should support forward erroring with a clear message

Suggested package:

- `internal/store/migrate`

## Testing Strategy

Testing is part of the design, not an afterthought.

### Unit Tests

Focus on:

- domain invariants
- item validation
- lease semantics
- status transitions
- sync state classification
- help registry completeness
- renderer output shapes

### Golden Tests

Use golden files for:

- `aj --help`
- per-command help
- workflow output
- prompt output
- compact item rendering

This is especially important because the help system is a product requirement.

### Store Tests

Use temp directories to test:

- init
- event append and meta rewrite
- lock behavior
- artifact writes
- cache rebuild
- recovery from partial write scenarios

### Jira Adapter Tests

Use a fake Jira server based on `httptest`.

Test:

- import
- export
- sync with remote-only changes
- sync with local-only changes
- conflict detection
- auth failure and rate limit handling

### End-To-End CLI Tests

Run the real compiled binary against temp repos.

Test flows:

- create and list work
- claim and release
- block and unblock
- handoff
- import Jira then sync
- help discovery commands

### Property-Style Tests

Use randomized tests where helpful for:

- ID generation collisions
- event replay consistency
- order-independent search behavior where applicable

## Observability And Debugging

Keep v1 simple.

Recommended:

- `--verbose` for extra operational details
- structured debug logs only when enabled
- no background daemon

Avoid noisy logs by default.

## Security And Safety

Rules for v1:

- never store Jira tokens in repo files
- sanitize remote text before terminal rendering
- require explicit `--resolve` for sync conflicts
- require explicit `--force` for lease override
- prefer `--dry-run` on Jira mutation commands

## Recommended Build Order

### Milestone 1: Bootstrapping

- initialize Go module
- root command and `aj init`
- config loading
- file layout creation
- lock scaffolding

### Milestone 2: Local Work Tracking

- `new`, `ls`, `show`, `update`, `done`
- item and event persistence
- compact renderers
- SQLite indexing for listing

### Milestone 3: Coordination

- `take`, `release`
- lease expiry logic
- `block`, `unblock`
- `handoff`
- `next`, `inbox`, `changes`

### Milestone 4: Discoverability

- structured help registry
- `aj help`, `aj commands`, `aj workflows`, `aj examples`, `aj glossary`

### Milestone 5: Jira Integration

- Jira config loading

- `--dry-run`

- migrations
- partial-write recovery

If we want the quickest path to a usable prototype, build these first:

1. `aj init`
2. `aj new`
3. `aj ls`
4. `aj show`
5. `aj update`
6. `aj take`
7. `aj block`
8. `aj done`
9. `aj --help`
10. `aj help <command>`

That gives us a real local workflow before Jira complexity arrives.

## Risks And Mitigations

### Risk: git merge pain on shared items

Mitigation:

- one event per file
- compact item snapshots
- item-level locks locally
- keep long notes out of `meta.toml`

### Risk: help quality degrades as commands grow

Mitigation:

- structured help registry
- golden help tests
- completeness test that every command is registered

### Risk: Jira sync becomes too magical

Mitigation:

- explicit link model
- explicit field ownership
- `sync_state`
- no silent conflict resolution

### Risk: cache and canonical files drift

Mitigation:

- treat SQLite as disposable
- update cache synchronously after writes
- rebuild on schema mismatch
